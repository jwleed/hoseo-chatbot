# MySQL 저장 기능 역할 정리

## 전체 흐름에서 MySQL의 위치

```
[프론트] ↔ [Spring Boot :8080] ↔ [MySQL]
                    ↓
             [AI 서버 :RAG]
```

Spring Boot가 AI 서버에 질문을 중계하는 역할이라면,
MySQL은 **그 과정에서 생기는 모든 데이터를 쌓아두는 곳**이다.

---

## 테이블별 역할

### users — 사용자 식별

| 역할 | 설명 |
|---|---|
| 무엇을 저장하나 | 앱에서 전달한 `userId` (deviceId) |
| 언제 만들어지나 | 처음 채팅 요청이 왔을 때 자동 생성 |
| 왜 필요한가 | 채팅방, 키워드 등 모든 데이터의 주인을 특정하기 위해 |

> 현재 ChatServiceImpl에서 메시지 저장 로직이 미구현이라 아직 실제로 생성되지 않음.
> 저장 로직 추가 시 `UserRepository.findByDeviceId()`로 조회 후 없으면 자동 생성 예정.

---

### chat_rooms — 채팅방

| 역할 | 설명 |
|---|---|
| 무엇을 저장하나 | sessionId, 채팅방 제목(첫 질문 앞 30자), 생성/수정 시각 |
| 언제 만들어지나 | 새 sessionId로 첫 질문이 왔을 때 |
| 왜 필요한가 | 대화를 묶어서 히스토리에서 "이전 대화" 단위로 보여주기 위해 |

사이드바에서 이전 대화 목록(`GET /api/history/{deviceId}`)을 불러올 때 이 테이블을 읽는다.
`updated_at`을 기준으로 최신순 정렬.

---

### chat_messages — 채팅 메시지

| 역할 | 설명 |
|---|---|
| 무엇을 저장하나 | role(`user`/`assistant`), 메시지 내용, 시각 |
| 언제 만들어지나 | 질문 시 USER 저장, SSE `[DONE]` 시점에 ASSISTANT 저장 |
| 왜 필요한가 | 이전 대화 클릭 시 메시지 목록을 다시 보여주기 위해 |

특정 채팅방 상세 조회(`GET /api/history/{deviceId}/{chatRoomId}`)가 이 테이블을 읽는다.
생성 시각 오름차순으로 정렬해서 대화 흐름을 재현.

> 현재 ChatServiceImpl에서 저장 로직 미구현. 히스토리가 쌓이려면 이 부분이 먼저 완성돼야 함.

---

### faqs — 자주 묻는 질문

| 역할 | 설명 |
|---|---|
| 무엇을 저장하나 | 카테고리, 질문 텍스트, 정렬순서, 클릭수, 활성여부 |
| 언제 만들어지나 | 앱 시작 시 `FaqDataInitializer`가 최초 1회 자동 삽입 |
| 왜 필요한가 | 채팅 시작 화면에 질문 버튼으로 뿌려주기 위해 |

프론트 동작 흐름:
```
앱 진입
  → GET /api/faq (카테고리별 질문 목록 조회)
  → 화면에 버튼으로 표시
  → 클릭
  → POST /api/chat/ask (AI에게 질문)
```

`view_count`는 FAQ 버튼 클릭 시마다 +1. 나중에 자주 클릭되는 질문을 상단에 노출하는 용도로 활용 가능.
`is_active = false`는 삭제 대신 숨김 처리 (soft delete).

---

### notices — 공지사항

| 역할 | 설명 |
|---|---|
| 무엇을 저장하나 | 공지 ID, 제목, 날짜, URL, 카테고리, 대상, 주관 부서 |
| 언제 만들어지나 | AI 서버가 `POST /api/notices/new` 호출 시 |
| 왜 필요한가 | 공지 알림 기능을 위한 데이터 축적 |

AI 서버(`incremental_notice_update.py`)가 매일 10:00~15:00 사이 30분 간격으로 새 공지를 크롤링해서 이 테이블에 쌓는다.

중복 방지 로직:
- `notice_id`가 있으면 → `notice_id`로 중복 체크
- `notice_id`가 없으면 → `title + date` 조합으로 중복 체크

나중에 사용자가 등록한 키워드와 비교해서 FCM 알림을 보내는 기능(5주차)의 원천 데이터가 된다.

---

## 현재 저장이 실제로 동작하는 것 vs 미구현

| 테이블 | 저장 동작 | 조회 동작 |
|---|---|---|
| users | ❌ 미구현 | ❌ |
| chat_rooms | ❌ 미구현 | ✅ API 있음 (데이터 없음) |
| chat_messages | ❌ 미구현 | ✅ API 있음 (데이터 없음) |
| faqs | ✅ 앱 시작 시 자동 삽입 | ✅ 동작 중 |
| notices | ✅ AI 서버 호출 시 저장 | — |

히스토리 기능을 실제로 쓰려면 **ChatServiceImpl에서 메시지 저장 로직**을 먼저 구현해야 한다.
